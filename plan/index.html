The state contract · what the UI is allowed to read

A snapshot the renderer consumes

{
  tick: 1843,
  hero: { x, y, vx, vy, facing,
          hp, hpMax, shield, shieldMax, alive },
  enemies: [
    { id, type:"ranged", tier:"hard",
      x, y, vx, vy, facing, hp, hpMax,
      state:"approach"|"bounce"|"stuck",
      hitFlash:0 }
  ],
  projectiles: [{ x, y, vx, vy, team }],
  effects: [   // gameplay effects only
    { kind:"blackhole", x, y, r, ttl } ],
  econ: { gold, xp, wave, killCount }
}

Why each field exists

FieldUI uses it to…
vx,vy / facingorient shapes, lean motion, interpolate between ticks
statepick anim: approaching vs bouncing vs stuck
hitFlashtrigger a UI-only spark burst (count not stored in sim)
hp/shieldhealth rings, damage shake
tiercolor lookup only — no rule logic in UI

Rule: if the UI needs a number to draw a frame, it's in the state. If it's decorative (smoke trail, screen flash), the UI invents it and forgets it on reload.
