ARENA. Dashboard Architecture Enemies Skills Abilities Upgrades Ideas

An upgrade system, layered to last forever.

A plan to adopt the proven progression skeleton of The Tower — Idle Tower Defense (Tech Tree Games) on top of what we already have. We don't reinvent: our unified gold/cores upgrade model already is The Tower's cash/coins workshop. This plan adds the missing layer that makes that game's progression endless — Labs (ceilings & scaling) — plus the upgrade breadth, enhancements, perks, and "tricks" that sit on top of it, and ties enemy scaling and cost to all of it so balance holds at every depth.

TL;DR — what we keep, what we add

Keep (already built): the dual-currency unified upgrade list (gold in-run = The Tower's cash; cores out-of-run = coins), attack/defense/economic tabs, cards-with-stars, tiers that double difficulty, milestones, deterministic sim + offline catch-up.

Add (this plan): a Lab layer that raises caps and multiplies scaling (the engine of endless growth), a much wider upgrade roster with real combat mechanics (multishot, rapid-fire, bounce, super-crit, rend, wall, thorns), Workshop Enhancements, Perks, discounts & respec, the ultimate-weapon / module / bot trick layer (reusing abilities.js), an expanded enemy roster, and a balance model that pins enemy HP/coins to the player's ceiling so the "wall" always lands where we want it.

3 · The core idea: ceilings vs fill

This is the single most important mechanic to internalise, and the one thing our game is missing. It's why The Tower never runs out of progression.

Workshop = fill

Cheap, fast, frequent. You buy hundreds of levels per run with cash, and a permanent stack with coins. But every upgrade has a level cap. Workshop fills the bar up to that cap — it can never exceed it.

In our code today: boughtOf() = min(max, perm+run). The max is the bar.

Labs = ceiling + slope

Slow, expensive, real-time-gated. A lab does one of two things: raise the cap (lets workshop fill higher) or steepen the curve / add a flat multiplier (every workshop level is worth more). Labs are where long-term power actually comes from.

The change: max stops being a constant and becomes baseMax + labCapBonus(stat), and computeStats multiplies by labScalingMult(stat).

Why two layers instead of one big number? Pacing. Workshop gives fast dopamine every few seconds; labs give a slow, planned, real-time investment that survives a death. The cap is the leash: it stops players from dumping a lucky cash spike into one stat and trivialising the run, and it turns "what do I research next" into the real meta-game. Enemy scaling is tuned against the cap, not the current level — so the cap is also our primary balance knob.

4 · Labs — the scaling / ceiling engine new system

A new meta system that lives entirely outside the deterministic run sim (it advances on the wall clock, in menus and while offline), so it never threatens replay determinism.

Data shape

// sim/labs.js — pure data + pure helpers, mirrors skills.js style
A.LABS = [
  // id, category, what it boosts, per-level effect, cost & time curves, cap, milestone gate
  { id:'dmgCap',     cat:'attack',  kind:'cap',   target:'rangedDamage', per:200,
    coin:curve(50,1.15), time:curve(60,1.18), max:60, gate:{tier:1,wave:30} },
  { id:'dmgScale',   cat:'attack',  kind:'scale', target:'rangedDamage', per:0.02,
    coin:curve(80,1.16), time:curve(120,1.2), max:50, gate:{tier:1,wave:60} },
  { id:'rateCap',    cat:'attack',  kind:'cap',   target:'attackSpeed',  per:20,  ... },
  { id:'gameSpeed',  cat:'utility', kind:'special',target:'gameSpeed',   per:0.1, max:25, ... },
  { id:'coinScale',  cat:'utility', kind:'scale', target:'coinsPerKill', per:0.05, ... },
  { id:'labSpeed',   cat:'utility', kind:'special',target:'labTime',     per:-0.02, ... },
  // ...one or two labs per workshop stat, plus economy/meta labs
];

// meta gains:
meta.labs   = {};        // { labId: level }   — completed research
meta.labSlots = 1;       // concurrent research slots (buy more with gems)
meta.research = [];      // active: [{ id, endsAt }]  endsAt = wall-clock ms
meta.labSpeedBoost = null;// { mult, until } from spending cells

How it works

  • Cap labs raise an upgrade's effective max: cap(id) = base.max + Σ per·level over cap-labs targeting it. boughtOf() already clamps to max — we just make max a function.
  • Scale labs add a multiplier applied in computeStats: ×(1 + Σ per·level).
  • Special labs drive things outside the stat block: game speed, lab speed, extra perk choices, coin-decay softening, enhancement unlocks.
  • Real-time research: starting a lab pushes {id, endsAt:now+time(level)·labSpeedMult} into meta.research if a slot is free, and deducts coins. On every menu load and tick we reconcile finished research (now ≥ endsAt → meta.labs[id]++). This is independent of the run RNG, so it's safe to advance from wall-clock deltas — see Architecture's offline section, but note labs use real time, not sim ticks.
  • Cancel = refund: cancelling an in-progress lab refunds its coins and remembers progress (green bar), exactly like The Tower's "switch research".
  • Slots: start at 1, buy up to 5 with gems (escalating price). More slots = research throughput, a major gem sink.
  • Speed-ups: spend cells (a farmed currency, see §9) for a timed 1.5×–5× research multiplier.
  • Gating: each lab unlocks only after its milestone (gate:{tier,wave}), reusing our milestone system so progression is paced.

Cost & time discipline (so it lasts thousands of levels)

coinCost(n) = round(base · 1.15..1.2n)     // steep — coins compound, so must this
timeSec(n)  = round(base · 1.18..1.22n)    // minutes → hours → days at the top
effectiveTime = timeSec(n) · (1 − labSpeedLab·k) · (1/activeSpeedBoost)

Early labs finish in minutes (keeps day-one players engaged); top labs take real days, which is the entire point of an idle game's retention curve. Time — not just coins — is the gate.

6 · Enhancements, discounts & respec

Workshop Enhancements late-game multiplier

A second, multiplicative pass over the same three categories, gated behind a deep milestone (our analogue of T12 W60). Each enhancement bought in a category unlocks the next in that category — a self-pacing tech tree. Mechanically identical to an upgrade row but its value enters computeStats as a × (1 + enhLevel·per) rather than a flat add, so it compounds with labs and cards. Stored as meta.enh = { id: level }.

Discounts

Three utility upgrades — Workshop Attack/Defense/Utility Discount — that reduce the coin price of permanent upgrades in their category by a %. Cleanly modeled as a multiplier on permCost(). A soft pressure valve that keeps deep coin costs reachable.

Respec

A button that refunds all coins spent on permanent workshop levels (optionally for a gem fee), zeroes meta.perm, and returns the coins. Trivial given our model (perm levels and prices are pure functions). Lets players re-pour into freshly cap-raised stats — the natural companion to labs.

7 · Perks — the in-run draft new system

From a wave threshold, offer 1-of-3 powerful run modifiers; chosen perks live in state.run.perks and feed computeStats as flats/mults. The wave gate grows as you take more, so perks stay precious all run.

A.PERKS = [
  { id:'dmgMult',   roll:()=>({mult:{rangedDamage:0.35}}) },
  { id:'rateMult',  roll:()=>({mult:{attackSpeed:0.30}}) },
  { id:'hpMult',    roll:()=>({mult:{maxHp:0.40}}) },
  { id:'freeUp',    roll:()=>({flat:{freeUpgradeChance:0.10}}) },
  { id:'gameSpeed', roll:()=>({special:{gameSpeed:0.25}}) },     // "more coins/min"
  { id:'waveReq',   roll:()=>({special:{perkWaveReq:-0.20}}) },  // the must-pick
  { id:'ultimate',  roll:(owned)=>grantRandomUnownedUltimate(owned) },
  // trade-offs: big up / small down, like The Tower's "trade-off" perks
];

// offer logic
nextPerkWave = 200; step = 50;     // after each pick: nextPerkWave += step·(1+count/…)
onWaveStart(n): if n >= nextPerkWave && !state.run.perkOffer:
  state.run.perkOffer = pick3(rng, ownedPerks);   // deterministic via run rng — pauses for choice

Because perk offers are drawn from the seeded run RNG, offline catch-up and replays stay deterministic; the UI just surfaces the pending offer when the player is watching, and an auto-pick rule resolves it during pure offline simulation.

New economy rules (in core.js cleanup / wave hooks)

  • Coins-per-kill ceiling + scaling: base coin value per enemy type, multiplied by a capped coinsPerKill bonus and lifted by a coin lab — so income scales but never explodes.
  • Coin decay: an enemy that survives > N waves loses 50% of its coin value — kills the "kite one wave forever" exploit and forces forward progress.
  • Interest: at each wave start, add min(maxInterest, interestRate · bankedCash) — rewards saving for big buys, a classic idle-TD hook.
  • Cash vs coins split tuning: cash income is generous (you should buy lots in-run); coin income is stingy (permanent power is earned slowly). The ratio is the long-term pacing dial.

10 · Enemy scaling & the balance loop

The whole point of the upgrade stack is to outrun an exponential enemy curve. Balance is not a table of numbers — it's pinning enemy growth to the ceiling the player can reach, so the wall always lands at the intended depth.

Keep the curve shape we have, retune the constants

enemyHP(wave,tier,enemyTier) = baseHP
   · waveStr(wave)            // today: linear 1+0.08·(n−1)  →  switch to mild exponential
   · TIERS[enemyTier].stat    // weak/avg/hard/elite class multiplier
   · tierDifficulty(tier)     // 2^(tier−1): each game-tier doubles effective wave

Today waveStr is linear (1+0.08·(n−1)). The player's power is multiplicative (damage × rate × crit × labs × cards…), which compounds far faster than a line — so right now the player eventually trivialises every wave. To create a real, recurring wall we move enemy HP to gently exponential (e.g. 1.025n blended with the linear term) so it tracks the player's compounding curve. The tier doubling stays as the coarse gate.

The balance identity we tune against

Wall depth ≈ where player ceiling DPS = enemy HP growth.
playerCeilingDPS(stat caps) ← set by labs (caps + scaling). enemyHP(wave) ← set by the curve above. We choose the lab cap/scaling constants so that, with maxed-for-your-progress labs, the wall sits ~50–100 waves past your current best. That gap is the carrot. Cores/coins income sets how many runs it takes to buy the labs that move the wall — i.e. the session count between walls.

11 · The full stat-composition formula

Every layer in one expression. This is what computeStats() grows into. Order matters: additive sources gather first, then multiplicative layers compound.

level   = min( cap(id), perm[id] + run[id] )          // cap(id)=baseMax + labCapBonus(id)
base    = UPGRADES[id].value(level)

effectiveStat(id) =
    ( base
      + Σ card.flat(id)
      + Σ perk.flat(id)
      + Σ module.flat(id) )
    · ( 1 + Σ labScale(id) )           // labs: the slope
    · ( 1 + Σ enhancement(id) )        // workshop enhancements: 2nd tier
    · Π ( 1 + card.mult(id) )
    · Π ( 1 + perk.mult(id) )
    · Π ( 1 + module.mult(id) )

// globals applied outside the per-stat block:
gameSpeed  = clamp( 1 + Σ labGameSpeed + Σ perkGameSpeed , 1, MAX )   // scales sim time → coins/min
permCost   = base.core.cost(n) · (1 − discount[cat])
coinReward = baseCoin · min(coinCap, 1+coinsPerKill) · (1+coinLab) · decayFactor(age)

We already have the additive/multiplicative resolution for cards in computeStats (the flat/mult split). Labs, enhancements, perks and modules slot into the same resolver as additional flat/mult/scale sources — so this is an extension of existing code, not a rewrite.

13 · Phased build order

Each phase is independently shippable and leaves the game playable. Phase 1 is the keystone — it delivers the "scaling & ceilings" the brief is really asking for.

Phase 1 — Dynamic caps + Labs MVP the keystone

Make max dynamic; add sim/labs.js with ~8 starter labs (damage cap+scale, attack-speed cap, health cap+scale, game speed, coin scale, lab speed); real-time research with offline reconciliation; a Lab menu tab. Retune enemy waveStr toward mild exponential so a wall actually exists.

Touches: sim/labs.js (new), sim/skills.js (cap(), labScale in computeStats), sim/waves.js (curve), sim/state.js + meta defaults, index.html (research reconcile on load/tick), hud/hud.js (Lab tab).

Phase 2 — Workshop breadth + Utility tab

Add the new A.UPGRADES rows. Pure-data ones (cash bonus, interest, free upgrades, defense absolute) first; then the combat-mechanic ones (multishot, bounce, rend, wall, thorns, rapid fire) with their sim behavior. Rename economic→utility.

Touches: sim/skills.js, sim/core.js, sim/projectiles.js, renderers/canvas2d/renderer.js (wall bar, bounce/rend FX).

Phase 3 — Economy rules

Coin-per-kill ceiling + scaling, coin decay, interest, cash bonus. Split gems/cells out of tokens and give them sources. This makes the lab economy actually pace correctly.

Touches: sim/core.js (cleanup/wave hooks), sim/enemies.js (drop tables), meta.

Phase 4 — Enhancements, discounts, respec

Second multiplier tier behind a deep milestone; per-category discount upgrades; coins respec. Small, high-impact, reuses Phase-1 plumbing.

Touches: sim/skills.js, hud/hud.js.

Phase 5 — Perks

sim/perks.js, wave-gated 1-of-3 draft drawn from run RNG, auto-pick rule for offline, growing wave requirement, perk effects into computeStats.

Touches: sim/perks.js (new), sim/core.js, sim/skills.js, hud/hud.js (draft overlay).

Phase 6 — Enemy roster + balance pass

Add fast/tank/protector/splitter/vampire rows & behaviors; tie cell drops; do the full balance pass pinning enemy HP to lab ceilings (§10). Boss cadence → module/gem drops.

Touches: sim/registries.js, sim/enemies.js, sim/core.js, sim/waves.js.

Phase 7 — Trick layer

Ultimate Weapons (promote abilities.js to a slot system) → Bots → Modules → Guardian, in that order. Each is opt-in build identity on top of a complete loop.

Touches: sim/abilities.js, sim/modules.js (new), sim/state.js (bots), HUD.

14 · Risks & open questions

  • Determinism vs real-time labs. Labs must advance on wall-clock and stay out of the seeded run sim, or offline replay breaks. Decision: labs are meta-state reconciled on load/menu tick, never inside Sim.step. (Game-speed does enter the sim and must be folded into the fixed-timestep + offline catch-up carefully.)
  • Curve retune is a real-balance change. Switching waveStr to exponential will invalidate current pacing; needs a sweep with the existing headless harness to place the first wall.
  • Currency identity. Should cores stay one currency that funds both permanent upgrades and labs (simpler), or split coins/cores? Recommend: cores = coins (fund both); tokens → gems (premium/slots); add cells (farmed). Worth a quick confirm before Phase 1.
  • Scope of Modules. Modules are the heaviest system and least essential to the core loop — keep them last and time-boxed.
  • UI density. The HUD will grow a lot (labs, enhancements, perks, modules). Plan tabbed menus early so it doesn't become unmanageable.
