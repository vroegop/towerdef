ARENA. Dashboard Architecture Enemies Skills Abilities Upgrades Ideas

Our own game, borrowing structure, not identity.

A plan that takes the proven progression structure of The Tower — Idle Tower Defense — but not its theme, names, or feel. The sim is an abstract growth engine; the UI decides what everything is called and how it looks. We want a unique game, not a clone. This covers only the four layers we actually want — upgrades, labs, cards, and (later) ultimate weapons — and ties enemy scaling and cost to them so balance holds at every depth.

TL;DR — what we keep, what we add

Keep (already built): the dual-currency unified upgrade list (an in-run currency + a persistent one), attack/defense/utility tabs, cards-with-stars, tiers, milestones, deterministic sim + offline catch-up.

Add (this plan): a Lab layer that raises caps and multiplies scaling (the engine of endless growth), a wider upgrade roster with real combat mechanics (multishot, rapid-fire, bounce, super-crit, rend, wall, thorns), a bigger card roster, and — later — ultimate weapons (reusing abilities.js), all tied to an enemy-scaling / balance model. Explicitly out of scope: enhancements, perks, modules, bots, guardian.

Design principles & scope

Abstract sim, expressive UI. The simulation only knows neutral numbers (a stat id, a value, a cost). What a stat is called, how it looks, and the theme of the whole game live entirely in the UI — so we can re-skin or rename anything without touching the engine. We borrow The Tower's structure; the identity stays ours.

In scope: upgrades labs cards ultimate weapons · later

Out of scope — not built, not planned around: enhancements · perks · modules · bots · guardian.

1 · The progression structure we borrow

Only the layers we want — each an orthogonal source of power that multiplies the others. Currencies here are described by role, not name: the UI will name them. (The Tower's own enhancements / perks / modules / bots / guardian are deliberately left out.)

3 · The core idea: ceilings vs fill

This is the single most important mechanic to internalise, and the one thing our game is missing. It's why The Tower never runs out of progression.

Upgrades = fill

Cheap, fast, frequent. You buy hundreds of levels per run with the in-run currency, and a permanent stack with the persistent one. But every upgrade has a level cap. Upgrades fill the bar up to that cap — they can never exceed it.

In our code today: boughtOf() = min(max, perm+run). The max is the bar.

Labs = ceiling + slope

Slow, expensive, real-time-gated. A lab does one of two things: raise the cap (lets workshop fill higher) or steepen the curve / add a flat multiplier (every workshop level is worth more). Labs are where long-term power actually comes from.

The change: max stops being a constant and becomes baseMax + labCapBonus(stat), and computeStats multiplies by labScalingMult(stat).

Why two layers instead of one big number? Pacing. Upgrades give fast dopamine every few seconds; labs give a slow, planned, real-time investment that survives a death. The cap is the leash: it stops players from dumping a lucky cash spike into one stat and trivialising the run, and it turns "what do I research next" into the real meta-game. Enemy scaling is tuned against the cap, not the current level — so the cap is also our primary balance knob.

4 · Labs — the scaling / ceiling engine new system

A new meta system that lives entirely outside the deterministic run sim (it advances on the wall clock, in menus and while offline), so it never threatens replay determinism.

Data shape

// sim/labs.js — pure data + pure helpers, mirrors skills.js style
A.LABS = [
  // id, category, what it boosts, per-level effect, cost & time curves, cap, milestone gate
  { id:'dmgCap',     cat:'attack',  kind:'cap',   target:'rangedDamage', per:200,
    coin:curve(50,1.15), time:curve(60,1.18), max:60, gate:{tier:1,wave:30} },
  { id:'dmgScale',   cat:'attack',  kind:'scale', target:'rangedDamage', per:0.02,
    coin:curve(80,1.16), time:curve(120,1.2), max:50, gate:{tier:1,wave:60} },
  { id:'rateCap',    cat:'attack',  kind:'cap',   target:'attackSpeed',  per:20,  ... },
  { id:'gameSpeed',  cat:'utility', kind:'special',target:'gameSpeed',   per:0.1, max:25, ... },
  { id:'coinScale',  cat:'utility', kind:'scale', target:'coinsPerKill', per:0.05, ... },
  { id:'labSpeed',   cat:'utility', kind:'special',target:'labTime',     per:-0.02, ... },
  // ...one or two labs per upgrade stat, plus economy/meta labs
];

// meta gains:
meta.labs   = {};        // { labId: level }   — completed research
meta.labSlots = 1;       // concurrent research slots (buy more with gems)
meta.research = [];      // active: [{ id, endsAt }]  endsAt = wall-clock ms
meta.labSpeedBoost = null;// { mult, until } from spending cells

How it works

  • Cap labs raise an upgrade's effective max: cap(id) = base.max + Σ per·level over cap-labs targeting it. boughtOf() already clamps to max — we just make max a function.
  • Scale labs add a multiplier applied in computeStats: ×(1 + Σ per·level).
  • Special labs drive things outside the stat block: game speed, lab speed, coin-decay softening, extra card slots.
  • Real-time research: starting a lab pushes {id, endsAt:now+time(level)·labSpeedMult} into meta.research if a slot is free, and deducts coins. On every menu load and tick we reconcile finished research (now ≥ endsAt → meta.labs[id]++). This is independent of the run RNG, so it's safe to advance from wall-clock deltas — see Architecture's offline section, but note labs use real time, not sim ticks.
  • Cancel = refund: cancelling an in-progress lab refunds its coins and remembers progress (green bar), exactly like The Tower's "switch research".
  • Slots: start at 1, buy up to 5 with gems (escalating price). More slots = research throughput, a major gem sink.
  • Speed-ups: spend cells (a farmed currency, see §7) for a timed 1.5×–5× research multiplier.
  • Gating: each lab unlocks only after its milestone (gate:{tier,wave}), reusing our milestone system so progression is paced.

Cost & time discipline (so it lasts thousands of levels)

coinCost(n) = round(base · 1.15..1.2n)     // steep — coins compound, so must this
timeSec(n)  = round(base · 1.18..1.22n)    // minutes → hours → days at the top
effectiveTime = timeSec(n) · (1 − labSpeedLab·k) · (1/activeSpeedBoost)

Early labs finish in minutes (keeps day-one players engaged); top labs take real days, which is the entire point of an idle game's retention curve. Time — not just coins — is the gate.

6 · Ultimate weapons later

The one “special” layer we keep — and only after upgrades, labs and cards are solid. Our sim/abilities.js already models cooldown-driven effects (black hole, meteor, golden window), which is exactly the shape an ultimate weapon needs. The work is promoting those hard-coded effects into a small, unlockable, upgradable slot system.

7 · Economy — currencies & rules

We already have the currencies; we mainly need to add the missing rules (decay, interest, ceilings) and one farmed currency to feed labs. Names below are roles — the UI titles them.

RoleOursLifecycleAction

New economy rules (in core.js cleanup / wave hooks)

  • Coins-per-kill ceiling + scaling: base coin value per enemy type, multiplied by a capped coinsPerKill bonus and lifted by a coin lab — so income scales but never explodes.
  • Coin decay: an enemy that survives > N waves loses 50% of its coin value — kills the "kite one wave forever" exploit and forces forward progress.
  • Interest: at each wave start, add min(maxInterest, interestRate · bankedCash) — rewards saving for big buys, a classic idle-TD hook.
  • Cash vs coins split tuning: cash income is generous (you should buy lots in-run); coin income is stingy (permanent power is earned slowly). The ratio is the long-term pacing dial.

8 · Enemy scaling & the balance loop

The whole point of the upgrade stack is to outrun an exponential enemy curve. Balance is not a table of numbers — it's pinning enemy growth to the ceiling the player can reach, so the wall always lands at the intended depth.

Keep the curve shape we have, retune the constants

enemyHP(wave,tier,enemyTier) = baseHP
   · waveStr(wave)            // today: linear 1+0.08·(n−1)  →  switch to mild exponential
   · TIERS[enemyTier].stat    // weak/avg/hard/elite class multiplier
   · tierDifficulty(tier)     // 2^(tier−1): each game-tier doubles effective wave

Today waveStr is linear (1+0.08·(n−1)). The player's power is multiplicative (damage × rate × crit × labs × cards…), which compounds far faster than a line — so right now the player eventually trivialises every wave. To create a real, recurring wall we move enemy HP to gently exponential (e.g. 1.025n blended with the linear term) so it tracks the player's compounding curve. The tier doubling stays as the coarse gate.

The balance identity we tune against

Wall depth ≈ where player ceiling DPS = enemy HP growth.
playerCeilingDPS(stat caps) ← set by labs (caps + scaling). enemyHP(wave) ← set by the curve above. We choose the lab cap/scaling constants so that, with maxed-for-your-progress labs, the wall sits ~50–100 waves past your current best. That gap is the carrot. Cores/coins income sets how many runs it takes to buy the labs that move the wall — i.e. the session count between walls.

9 · The stat-composition formula

Every layer we keep, in one expression. This is what computeStats() grows into. Order matters: additive sources gather first, then multiplicative layers compound.

level   = min( cap(id), perm[id] + run[id] )          // cap(id)=baseMax + labCapBonus(id)
base    = UPGRADES[id].value(level)

effectiveStat(id) =
    ( base + Σ card.flat(id) )
    · ( 1 + Σ labScale(id) )           // labs: the slope
    · Π ( 1 + card.mult(id) )           // cards: flat & mult, already implemented

// globals applied outside the per-stat block:
gameSpeed  = clamp( 1 + Σ labGameSpeed , 1, MAX )    // scales sim time → income/min
coinReward = baseCoin · min(coinCap, 1+coinsPerKill) · (1+coinLab) · decayFactor(age)

We already have the additive/multiplicative resolution for cards in computeStats (the flat/mult split). Labs slot into the same resolver as additional cap/scale sources — so this is an extension of existing code, not a rewrite. No perk, enhancement, or module terms: those layers don’t exist in our design.

11 · Phased build order

Each phase is independently shippable and leaves the game playable. Phase 1 is the keystone — it delivers the "scaling & ceilings". Ultimate weapons come last, after the basics.

Phase 1 — Dynamic caps + Labs MVP the keystone

Make max dynamic; add sim/labs.js with ~8 starter labs (damage cap+scale, attack-speed cap, health cap+scale, game speed, income scale, lab speed); real-time research with offline reconciliation; a Lab menu tab. Retune enemy waveStr toward mild exponential so a wall actually exists.

Touches: sim/labs.js (new), sim/skills.js (cap(), labScale in computeStats), sim/waves.js (curve), sim/state.js + meta defaults, index.html (research reconcile on load/tick), hud/hud.js (Lab tab).

Phase 2 — Upgrade breadth

Add the new A.UPGRADES rows. Pure-data ones (cash bonus, interest, free upgrades, defense %) first; then the combat-mechanic ones (super-crit, thorns, rend, multishot, bounce, rapid fire, wall) with their sim behaviour — each is one of the specs in §15.

Touches: sim/skills.js, sim/core.js, sim/projectiles.js, renderers/canvas2d/renderer.js (wall bar, bounce/rend FX).

Phase 3 — Economy rules

Kill-income ceiling + scaling, coin decay, interest, cash bonus. Give tokens (premium) and a new cells (farmed) currency real sources. This makes the lab economy pace correctly.

Touches: sim/core.js (cleanup/wave hooks), sim/enemies.js (drop tables), meta.

Phase 4 — Card roster growth

Data-only cards mapping to the new upgrade stats (Bramble/Volley/Ricochet/Sunder/…), each shipped with or just after the stat it references. Reuses the existing card system entirely.

Touches: sim/skills.js (A.CARDS, STAT2SIM), hud/hud.js (icons/labels).

Phase 5 — Enemy roster + balance pass

Add fast/tank/protector/splitter/vampire rows & behaviours; tie cell drops; the full balance pass pinning enemy HP to lab ceilings (§8).

Touches: sim/registries.js, sim/enemies.js, sim/core.js, sim/waves.js.

Phase 6 — Ultimate weapons later

Promote abilities.js to an unlockable A.ULTIMATES slot system with a small per-weapon upgrade tree, tuned by a few labs & cards. Opt-in flavour on top of a complete loop. (No modules, bots or guardian.)

Touches: sim/abilities.js, sim/skills.js, hud/hud.js.

12 · How to use these specs

Everything below the strategy is now an actionable backlog. Each feature is a self-contained work package — roughly one PR — written so it can be picked up in isolation without reading the others. The intent is that you skim a spec, look at its mockup or toggle it in the live sandbox, and give a thumbs-up / change-this before any code is written.

Each spec contains

Goal · the exact data row · the sim hook (file & function) · a numbered step list · headless acceptance criteria · and a visual (HUD mockup or sandbox toggle).

Status legend

How to give feedback

Reference a spec by its heading (e.g. “Thorns” or “Multishot”). Say build it, change the numbers, or not yet. I implement one spec per PR so each is easy to review and revert.

Super Crit

~1h · 1 file (sim/skills.js)

A crit can “crit again”: a chance for the crit multiplier to apply an extra time. Our damage roll already supports stacking crits — this just exposes the chance as an upgrade.

Data row (Attack)

{ id:'superCrit', tab:'attack', icon:'burst', label:'Super Crit', max:1000,
  value:(b)=> b*0.001, fmt:(b)=>(b*0.001*100).toFixed(1)+'%',   // up to +100%
  gold:curve(80,1.6), core:curve(25,1.0018) }

Sim hook — core.js · _rollDamage()

let k = Math.floor(st.critChance);
if (rng.next() < st.critChance - k) k++;
// NEW: on a crit, super-crit rolls an extra multiplier application
if (k > 0 && rng.next() < st.superCrit) k++;

Steps

  1. Add the row to A.UPGRADES and superCrit: U.superCrit.value(...) to computeStats.
  2. Add the one if in _rollDamage.
  3. Done — HUD lists it automatically (icon burst exists).

Acceptance

  • Headless: with critChance=1 and superCrit=1, every shot applies critMult².
  • With superCrit=0, damage is unchanged from today.

Defense % (multiplicative reduction)

~1h · 2 files

A percentage damage reduction that stacks after flat Armor — Armor subtracts, then Defense % scales the remainder. Together they cover early (flat) and late (%) game.

Data row (Defense)

{ id:'defPct', tab:'defense', icon:'shield', label:'Defense %', max:900,
  value:(b)=> Math.min(0.9, b*0.001), fmt:(b)=>(Math.min(0.9,b*0.001)*100).toFixed(1)+'%',
  gold:curve(50,1.6), core:curve(20,1.008) }   // caps at 90% DR

Sim hook — core.js · _hurtHero()

let amt = (amount - (st.armor||0));
amt *= (1 - (st.defPct||0));        // NEW
if (amt <= 0) return;

Steps

  1. Add row + defPct to computeStats.
  2. Multiply the post-armor amount in _hurtHero.
  3. Hard-cap value at 0.9 (done in value) so a hit always lands.

Acceptance

  • Armor 2 + Defense 50% vs a 12-dmg hit → (12−2)·0.5 = 5 taken.
  • Capped: even maxed, ≥10% of every hit lands (no invulnerability).

Lifesteal

½ day · 2 files

Heal the hero for a fraction of damage dealt — a sustain alternative to Regen that scales with offense.

Data row (Defense)

{ id:'lifesteal', tab:'defense', icon:'regen', label:'Lifesteal', max:500,
  value:(b)=> Math.min(0.25, b*0.0005), fmt:(b)=>(Math.min(0.25,b*0.0005)*100).toFixed(1)+'%',
  gold:curve(60,1.6), core:curve(15,1.0018) }

Sim hook — on damage dealt (projectiles.js impact & core.js lightning branch)

if (st.lifesteal) {
  hero.hp = Math.min(hero.hpMax, hero.hp + dealt * st.lifesteal);

Steps

  1. Add row + lifesteal to computeStats.
  2. Where a bullet/lightning applies damage, heal the hero (clamp to hpMax).

17 · Risks & open questions

  • Determinism vs real-time labs. Labs must advance on wall-clock and stay out of the seeded run sim, or offline replay breaks. Decision: labs are meta-state reconciled on load/menu tick, never inside Sim.step. (Game-speed does enter the sim and must be folded into the fixed-timestep + offline catch-up carefully.)
  • Curve retune is a real-balance change. Switching waveStr to exponential will invalidate current pacing; needs a sweep with the existing headless harness to place the first wall.
  • Currency identity. Should cores stay one currency that funds both permanent upgrades and labs (simpler), or split coins/cores? Recommend: cores = coins (fund both); tokens → gems (premium/slots); add cells (farmed). Worth a quick confirm before Phase 1.
  • UI density. The HUD will grow with labs and a wider upgrade/card roster. Plan tabbed menus early so it doesn't become unmanageable.
