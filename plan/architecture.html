Architecture

One rule above all others: the simulation never imports the renderer, and the renderer never mutates the simulation.

The three-layer split

SIM (core)
Pure logic. No DOM, no canvas, no Date.now() inside steps. Deterministic. Could run in a Worker or on a server unchanged.
STATE
A plain serializable snapshot. The only thing the two layers share. Saveable to disk verbatim.
UI (render)
Reads snapshots, draws shapes, owns decorative-only effects. The HUD calls upgrade actions straight into the sim.

Data flows SIM → STATE → UI for rendering; the only thing flowing back is the HUD's upgrade actions. The UI's decorative particles are a dead-end branch off STATE that never flows back — which is exactly why they can be thrown away on reload with no consequence.

Concrete file shape. Keep sim/ with zero browser imports (sim/core.js, sim/enemies.js, sim/skills.js, sim/rng.js) and ui/ on the other side (ui/renderer.js, ui/effects.js, ui/hud.js). A lint rule "sim/ may not import from ui/" enforces the boundary mechanically.

Folder structure (scaffolded)

The split above is now real on disk. sim/ has zero browser imports; everything else depends on it, never the reverse.

Why plain <script> + a global ARENA namespace instead of ES modules? ES module scripts fail under file:// (CORS), and the goal is double-click-to-open. The sim files still have no browser dependencies, so they remain portable to a Web Worker or a Node test runner — only the packaging is simple. Swapping to ESM + a bundler later is a mechanical change.
The renderer seam is the payoff. A renderer only reads a snapshot (draw(snapshot, alpha)) and never mutates state or holds rules. Drop a WebGL/Pixi/Three prototype in renderers/<name>/, point index.html at it, and compare feel side-by-side — same sim, different paint.

Deterministic fixed-timestep loop

Real time is messy — frames arrive every 16ms, 40ms, 200ms after a stall. If the sim advanced by "whatever time passed", two machines would diverge and offline catch-up would be impossible. So the sim only ever advances in fixed quanta (e.g. DT = 1/30s). The render loop accumulates real time and spends it in whole ticks.

const DT = 1/30;            // logical tick size, forever constant
let acc = 0, last = now();

function frame(t){
  acc += (t - last)/1000; last = t;
  let guard = 0;
  while (acc >= DT && guard++ < 8){   // guard caps catch-up after a stall
    sim.step(DT);                    // the ONLY way state changes
    acc -= DT;
  }
  renderer.draw(sim.snapshot(), acc/DT); // pass leftover for interpolation
  requestAnimationFrame(frame);
}
Why the acc/DT remainder matters for battery. Because render and sim are decoupled, you can run the sim at 30Hz but still draw buttery 60fps by interpolating positions between the last two sim states. Or drop the renderer to 30fps to save battery while the sim is unchanged. The two rates are independent knobs.

Determinism = seeded RNG, integer-ish math, no wall clock

"Deterministic" means step() is a pure function of (state, dt). Three disciplines make it so:

  • Seeded PRNG only. Never Math.random() in the sim — it's unseeded and unreproducible. Use a small LCG/xorshift whose seed lives in the state, so a saved game resumes the same fight.
  • No Date.now() / performance.now() inside a step. Time is the tick counter only.
  • Stable iteration order. Iterate arrays, not Object key order or Set insertion quirks, so the same inputs always produce the same outputs.
// rng.js — seed travels with the save file
function makeRng(seed){
  let s = seed >>> 0;
  return () => { s = (s*1664525 + 1013904223) >>> 0; return s/4294967296; };
}
Payoff: the live game and the offline catch-up call the identical step(). There is no second "offline engine" to keep in sync — the #1 source of idle-game exploits and bugs.

Offline simulation — battles continue while the screen is off

The app is not running in the background. Instead, on every resume we measure the gap and replay it.

1
On pause/close (the visibilitychange → hidden, or app suspend) write savedAt = Date.now() alongside the full sim state. The seed is part of the state, so the future is already determined.
2
On resume compute elapsed = Date.now() - savedAt (clamp to a sane cap, e.g. 12–24h, and guard against clock tampering by also tracking monotonic where available).
3
Replay ticks = floor(elapsed / DT) by calling step() in a tight loop with no rendering — millions of ticks per second because there's no canvas, no allocation churn, no rAF.
4
Stop at death. If hero.alive goes false at tick K, stop there. The reward summary reflects what was earned up to the moment of death — never beyond it.
5
Show a summary ("While you were away: 4,210 kills, +18,400 gold, died at 6h12m to an Elite boss") then hand the live loop a fresh, UI-effect-free state.
function catchUp(state, elapsedMs){
  const ticks = Math.min(Math.floor(elapsedMs/1000/DT), MAX_TICKS);
  const start = state.econ.snapshot();
  for (let i=0; i<ticks; i++){
    sim.step(DT);
    if (!state.hero.alive) break;     // sim stops at death
  }
  return diff(start, state.econ);        // the "while you were away" report
}
The speed knob: coarse-tick batching. Replaying 12h at 30Hz is ~1.3M ticks — fine for a shape sim. But if combat math gets heavy, add an adaptive coarse mode: once you're far from any decision boundary (no enemy near death, hero healthy), advance with a larger DT or use a closed-form "this wave deals X dps, hero sustains Y" approximation, then drop back to fine ticks near interesting events. Keep the coarse model conservative so it can never over-reward.
Run it in a Web Worker. Because the sim has no DOM deps, the catch-up loop (and even live ticking) can live in a Worker. The main thread stays free for paint, the UI never jank-freezes during a long replay, and you can show a progress bar driven by postMessage.

Events bridge the gap cleanly. The sim emits a tiny per-tick event log ({type:'hit', id, dmg, crit}). The UI reads it to spawn the right particles/numbers, then discards it. This keeps damage numbers accurate (sim-authored) while their animation stays UI-only — and on reload the UI simply starts with an empty effect pool.

Battery & performance discipline

  • Pause everything when hidden. visibilitychange → stop rAF and stop ticking; record savedAt. No work happens off-screen — the catch-up replaces it.
  • One canvas, no DOM per entity. Hundreds of shapes drawn with primitive calls beat hundreds of DOM nodes by orders of magnitude.
  • Avoid per-frame allocation. Reuse entity objects and particle pools; garbage collection pauses are the enemy of smoothness and battery.
  • Spatial grid for collisions only when counts grow. Circular hitboxes make it a cheap distance check; a uniform grid keeps it O(n) instead of O(n²).
  • Cap the renderer, not the sim. Offer 30/60fps render setting; the sim stays 30Hz so gameplay feel is identical.
  • Integer-friendly math & flat typed arrays later if profiling demands it — but only then.
