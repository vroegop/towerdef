Skills & Stats

A flat stat block, a single well-defined stacking formula, and upgrade curves expressed purely as data.

The hero stat block

Every skill ultimately edits one of these base numbers. Keeping them in one flat object means the sim reads them directly and saves/serializes them trivially.

const baseStats = {
  // survival
  health:100, healthRegen:1.0,        // hp, hp/sec
  shield:0,   shieldRegen:0, shieldDelay:3, // hp, hp/sec, sec after hit
  armor:0,                              // % damage reduction (diminishing)
  // offense
  meleePower:14, meleeRate:4,          // dmg, attacks/sec
  rangedPower:10,rangedRate:2,range:90,  // dmg, shots/sec, px
  critChance:0.05, critMult:1.5,
  // economy
  pickupRange:40,
  goldFind:1.0, xpGain:1.0,
};

One stacking formula for everything

Mixing flat and percentage bonuses naively creates balance chaos. Use one explicit order so every source is predictable and the offline sim computes the same effective value the live game shows.

// effective = (base + Σ flat) × (1 + Σ additive%) × Π (1 + mult%)
function effective(base, mods){
  let flat=0, add=0, mul=1;
  for(const m of mods){
    if(m.kind==='flat') flat += m.v;
    else if(m.kind==='add')  add  += m.v;   // stack additively (e.g. +10% +10% = +20%)
    else                     mul  *= (1+m.v); // stack multiplicatively (rarer, stronger)
  }
  return (base+flat) * (1+add) * mul;
}
Why three buckets? Flat = early-game meaningful, late-game irrelevant (self-balancing). Additive% = the bread-and-butter upgrades that stack linearly so they're easy to reason about. Multiplicative% = reserved for rare/prestige sources so they feel special and don't trivialize balance when combined.
Armor uses diminishing returns, not raw subtraction. A common, exploit-proof curve: reduction = armor / (armor + K) (e.g. K=100). 100 armor = 50% reduction, 300 = 75% — it asymptotes, so armor never reaches 100% and stacking stays meaningful without ever becoming invincibility.

Keystones are just modifiers with a mult bucket entry plus a flag the sim reads. They live in the same effective() pipeline — no special casing, so they're safe in the offline sim too.
